🔤 Tokenization — Step-by-Step Simulation

🔤 How Text Becomes Tokens (BPE · cl100k_base · tiktoken)

1 Raw characters — every letter, space and punctuation mark
The model never sees raw text — it starts by treating the input as a flat sequence of Unicode code points.
2 Byte-Pair Encoding (BPE) — merge frequent pairs into sub-words
BPE learns a vocabulary of ~100 000 sub-word units by iteratively merging the most frequent adjacent byte pairs. Common words become one token; rare words are split into pieces. This lets the model handle any language and new words gracefully.
3 Token IDs — each sub-word maps to an integer in the vocabulary
These integer IDs are the actual input to the neural network. The model learns a 1 536-dimensional embedding vector for each ID. That embedding is what enters the attention layers.
4 Positional encoding — order is injected into every embedding
Transformers process all tokens in parallel — no built-in notion of order. A positional signal (sinusoidal or learned RoPE) is added to each embedding so the model knows which token came first, second, etc.
BPE cl100k_base tiktoken ~100k vocab 1536-d embeddings RoPE positional

🔍 How Semantic Search Finds Relevant Chunks (OpenSearch kNN · Cosine Similarity)

1 Query embedding — the prompt is encoded into a 1 536-d vector (Bedrock Titan Embeddings)
Each dimension captures a semantic feature. The chart below shows a 32-d projection of the 1 536-d vector:
Query vector — 1 536 dimensions (shown as 32-d projection)
2 Knowledge-base chunks — stored in OpenSearch as pre-computed embedding vectors (S3 → Bedrock at ingest)
5 candidate document chunks from the index:
3 Cosine similarity scoring — dot product ÷ (|A| × |B|) for each chunk vs. the query
Formula: cos(θ) = (A · B) / (‖A‖ × ‖B‖) — range −1 to 1, higher = more similar
4 Top-K retrieval — highest-scoring chunks injected into the context window
Only chunks above the similarity threshold (default 0.80) are included. These are prepended to the LLM prompt as grounding context.

MCP vs RAG — Context Enrichment Approaches
🔌 MCP
Model Context Protocol
What it isStandardised protocol for LLMs to call external tools & data sources at runtime
How it worksAgent sends a structured tool-call; MCP server executes and returns live data
Data freshnessReal-time / live
LatencyHigher — waits for external call to complete
SetupMCP server per integration (Slack, DB, APIs, file systems…)
Cost driverAPI / compute cost per tool invocation
Best forActions, live data, write operations (create ticket, send email)
AWS serviceLambda + API Gateway + SQS
VS
📚 RAG
Retrieval-Augmented Generation
What it isInject relevant document chunks into the prompt via semantic vector search
How it worksQuery → embed → kNN search → top-K chunks prepended to context window
Data freshnessAs fresh as last ingest
LatencyLow — vector search is sub-10ms at scale
SetupIngest pipeline: chunk → embed → upsert into vector index
Cost driverEmbedding model + index storage + extra input tokens
Best forRead-only knowledge Q&A, document grounding, compliance citations
AWS serviceOpenSearch + S3 + Bedrock Titan Embeddings
✅ Use MCP when…
  • You need live / real-time data (stock prices, CRM records, weather)
  • The agent must write — create tickets, send messages, update DBs
  • Integrating with enterprise systems (ServiceNow, Salesforce, Jira)
  • Data changes frequently and staleness is unacceptable
✅ Use RAG when…
  • Large document corpus (policies, manuals, research reports)
  • Need citations and source grounding for compliance/audit
  • Low-latency responses are critical
  • Data is relatively static (updated weekly/monthly)
MCP Flow
1. Agent decides a tool is needed → emits tool-call JSON
↓
2. MCP client deserialises the call and routes to correct server
↓
3. MCP server executes (DB query, API call, file read…)
↓
4. Result returned as structured JSON → appended to context
↓
5. LLM re-invoked with tool results → final response
MCP Server Types
🗄️ Database — SQL/NoSQL query execution (read + write)
🌐 Web / API — REST/GraphQL calls to external services
📁 File System — read/write local or S3 files
📧 Messaging — send emails, Slack, Teams messages
🎫 Ticketing — ServiceNow, Jira, GitHub Issues
📊 Analytics — run queries against data warehouses
🔐 Auth/Identity — look up user profiles, permissions
RAG Ingest Pipeline
1. Raw document (PDF/DOCX/HTML) → S3
↓
2. Chunker splits into ~512-token overlapping windows
↓
3. Bedrock Titan Embeddings → 1 536-d vector per chunk
↓
4. Upsert vector + metadata into OpenSearch HNSW index
RAG Query Pipeline
1. User query → embed with same Titan model
↓
2. kNN cosine search → top-K chunks (threshold 0.80)
↓
3. Chunks prepended to prompt as grounding context
↓
4. LLM generates answer citing retrieved passages
RAG Tuning Knobs
Chunk size — 256 / 512 / 1024 tokens. Smaller = precise, larger = context-rich
Overlap — 10–20% overlap prevents context loss at boundaries
Top-K — number of chunks retrieved (typical: 3–10)
Similarity threshold — minimum cosine score to include (0.75–0.85)
Re-ranking — cross-encoder re-scores top-K before injection
Hybrid search — combine BM25 (keyword) + vector for best recall
Metadata filters — filter by date, document type, department

⚙️ Infrastructure Deep-Dive — Lambda vs EKS Deployment


  


    
    

      

        
λ

        

          
AWS Lambda

          
Serverless Functions

          
Think of Lambda like a vending machine — it sits idle, costs nothing, then instantly dispenses when you press a button. You only pay per use. But there's a warm-up time if it hasn't been used recently (cold start).

        

      

      
VS

      

        
⎈

        

          
Amazon EKS

          
Kubernetes Container Clusters

          
Think of EKS like a fleet of always-on workers — containers run 24/7, respond instantly with zero warm-up, and can handle long-running jobs. You pay for the servers whether busy or idle.
