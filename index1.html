System 2 Reasoning

Core Concept

Borrowed from cognitive psychology. System 1 = fast, instinctive (traditional bots). System 2 = slow, deliberate, logical. Atlas uses System 2 Гўв‚¬вЂќ it pauses, evaluates context, considers options, and plans before acting. This is why it handles edge cases that scripted bots fail on.

ГўЕЎ ГЇВёВЏ Interview Trap

MCP (Model Context Protocol) is not where reasoning happens. MCP is the integration layer for connecting external tools. Atlas does the reasoning. Don't confuse the two in interviews.

Module 03

Core Terminology

Every term you must know cold for developer interviews and Agentblazer Legend certification.

Agent

Foundation

A configured AI system in Agentforce that has a defined persona, set of topics, actions, and guardrails. An agent receives user input, reasons through it using Atlas, and produces outputs by executing actions. Think of it as an autonomous digital employee with a specific job role.

Topic

Building Block

A scoped domain of behavior within an agent. Each topic has a classification description (tells Atlas when to activate it), a scope (what the agent can do in this topic), and a set of actions. Example: an e-commerce agent might have topics for "Order Management", "Returns", and "Product Queries".

Action

Building Block

A specific task the agent can execute. Actions are registered tools Гўв‚¬вЂќ they can be Salesforce Flows, Apex classes, API callouts, prompt templates, or even other agents. Actions have input/output parameters that Atlas uses to pass data through the execution chain.

Instructions

Building Block

Natural language directives that shape agent behavior within a topic. Instructions define: the agent's persona, communication tone, what it can/cannot say, when to ask clarifying questions, and when to escalate. Well-written instructions are the difference between a reliable agent and a hallucinating one.

Guardrails

Safety

Runtime validation rules applied before a response is sent. Guardrails prevent the agent from: going out of scope, revealing sensitive data, taking destructive actions, or responding when it has insufficient grounded data. They run on every turn, not just at setup.

Subagent

Orchestration

A focused agent invoked by an orchestrator agent to handle a specialized domain. Subagents encapsulate their own topics, actions, and instructions. They accept context from the orchestrator, execute their task, and return structured output. This is the foundation of multi-agent architecture.

Agentforce Script

Advanced

A unified text-based language that expresses the entire behavior of an agent in one file Гўв‚¬вЂќ configuration, business logic, actions, and reasoning instructions. It blends natural language with deterministic rules, enabling hybrid reasoning that's both flexible and predictable.

Data Cloud / Data 360

Data Layer

The data virtualization layer that provides individualized context to agents. Agents query Data Cloud to ground their responses in real CRM records, customer profiles, order history, and unstructured assets (emails, PDFs, knowledge articles). Without Data Cloud grounding, agents hallucinate.

Model Context Protocol (MCP)

Integration

A universal integration standard (like a USB-C port for AI) that allows agents to securely connect to external tools, files, databases, and APIs without custom integration code. Salesforce shipped 60+ MCP tools in Spring '26. MCP is the transport layer; Atlas is the reasoning layer.

Prompt Template

AI Layer

A structured template used to inject CRM data, user context, and instructions into the LLM call. Prompt templates are configured in Prompt Builder and registered as actions. They control exactly what information is sent to the model and in what format Гўв‚¬вЂќ critical for grounding and accuracy.

Building Blocks Deep Dive

How Topics, Actions, and Instructions actually work together in Agent Builder Гўв‚¬вЂќ with real configuration examples.

Topics Гўв‚¬вЂќ the right way to configure them

ГўЕ“вЂ¦ Good topic classification description

"Use this topic when a customer asks to cancel, modify, or check the status of an existing order. Activate when the conversation contains order numbers, cancellation requests, or delivery inquiries."

ГўВќЕ’ Bad topic classification description

"Handle order things." Гўв‚¬вЂќ Too vague. Atlas will misroute or activate this topic incorrectly, causing wrong actions to fire on unrelated queries.

Actions Гўв‚¬вЂќ three types you must know

Flow Action

Autolaunched Flows registered in Agent Builder. Best for: data lookups, record updates, multi-step logic with decisions. Input/output variables become the action's parameters.

Apex Action

Apex methods annotated with @InvocableMethod. Best for: complex logic, external API callouts, transactional operations needing try-catch and rollback.

Prompt Action

Prompt templates from Prompt Builder. Best for: AI-generated summaries, personalized messages, classification tasks where you need the LLM to produce structured output.

Agent Action

Calling another agent as an action. This is the foundation of multi-agent orchestration. The called agent returns structured output consumed by the orchestrator.

Instructions Гўв‚¬вЂќ what to write

// Example topic instructions for an Order Cancellation topic

You are an order management specialist for Acme Corp.

Always:
- Collect the order number from the customer if not provided
- Verify eligibility before cancelling (within 24h, status = Processing)
- Confirm the cancellation with the customer before executing
- Provide a cancellation reference number after success

Never:
- Cancel orders that are already shipped or delivered
- Reveal internal order IDs or database record IDs
- Process refunds in this topic (use the Billing topic)

If eligibility check fails:
- Explain why the order cannot be cancelled
- Offer to connect the customer with a human specialist
- Use the escalation action to transfer to Omni-Channel queue

Architecture Patterns

Real architectural patterns used in enterprise Agentforce deployments.

Pattern 1 Гўв‚¬вЂќ Single Agent (standard)

One agent handles all topics for a single domain. Topics route internally. Best for: focused use cases like a customer service agent for a single product line. Easiest to build and maintain.

Code Examples

Real Apex and Flow patterns you'll build in production Agentforce deployments.

Example 1 Гўв‚¬вЂќ Apex Action for Order Eligibility Check

Apex
public class OrderCancellationAction {

    public class OrderInput {
        @InvocableVariable(label='Order Number' required=true)
        public String orderNumber;
    }

    public class OrderResult {
        @InvocableVariable(label='Is Eligible')
        public Boolean isEligible;

        @InvocableVariable(label='Order Id')
        public String orderId;

        @InvocableVariable(label='Reason')
        public String reason;
    }

    @InvocableMethod(
        label='Check Order Cancellation Eligibility'
        description='Returns eligibility status and reason for cancellation request'
    )
    public static List<OrderResult> checkEligibility(
        List<OrderInput> inputs
    ) {
        List<OrderResult> results = new List<OrderResult>();

        for (OrderInput inp : inputs) {
            OrderResult result = new OrderResult();

            // Query order record
            List<Order> orders = [
                SELECT Id, Status, CreatedDate
                FROM Order
                WHERE OrderNumber = :inp.orderNumber
                LIMIT 1
            ];

            if (orders.isEmpty()) {
                result.isEligible = false;
                result.reason = 'Order not found';
            } else {
                Order ord = orders[0];
                Boolean within24h = ord.CreatedDate >= System.now().addHours(-24);
                Boolean isProcessing = ord.Status == 'Processing';

                result.isEligible = within24h && isProcessing;
                result.orderId = ord.Id;
                result.reason = !within24h
                    ? 'Order is older than 24 hours'
                    : !isProcessing
                    ? 'Order status is not Processing'
                    : 'Eligible for cancellation';
            }
            results.add(result);
        }
        return results;
    }
}

Example 4 Гўв‚¬вЂќ Apex with try-catch for transactional safety

Apex
@InvocableMethod(label='Process Refund')
public static List<String> processRefund(List<String> orderIds) {
    List<String> results = new List<String>();

    for (String orderId : orderIds) {
        Savepoint sp = Database.setSavepoint();
        try {
            // 1. Update order status
            Order ord = [SELECT Id, Status FROM Order WHERE Id = :orderId];
            ord.Status = 'Refund Initiated';
            update ord;

            // 2. Call external payment API
            HttpResponse res = RefundAPIService.initiateRefund(orderId);
            if (res.getStatusCode() != 200) {
                throw new CalloutException('Payment API failed: ' + res.getBody());
            }

            results.add('SUCCESS:Refund initiated for order ' + orderId);

        } catch (Exception e) {
            // Rollback ALL changes if anything fails
            Database.rollback(sp);
            results.add('ERROR:' + e.getMessage());
        }
    }
    return results;
}
Module 07

Anti-Hallucination Techniques

The #1 production issue with Agentforce. Here are the root causes and specific fixes.

Root Cause 1 Гўв‚¬вЂќ No data grounding

The agent answers from the LLM's general knowledge instead of your actual business data. This is the most common cause.

Fix: Connect Data Cloud. Use vector search (RAG) to retrieve relevant records before generating a response. Add to instructions: "Only answer based on the data provided. If the answer is not in the retrieved context, say you don't know."

Root Cause 2 Гўв‚¬вЂќ Weak prompt templates

Vague instructions give the LLM room to fill gaps with invented content. Prompts that don't constrain the output format or data sources allow hallucination.

Fix: Add explicit constraints in every prompt template: specify the output format, the data source to use, and what to do when data is absent. Use Agentforce Script for deterministic, rule-based instructions alongside natural language.

Root Cause 3 Гўв‚¬вЂќ Stale knowledge base

Outdated knowledge articles return wrong information confidently. Agents have no way to know an article is outdated without explicit metadata.

Fix: Audit your knowledge base regularly. Set PublishStatus and LastModifiedDate filters in your Data Cloud ingestion to exclude articles older than a defined threshold. Scope the agent's knowledge search to relevant article categories only.

Root Cause 4 Гўв‚¬вЂќ No fallback instruction

When the agent finds no grounded data, it defaults to generating something plausible-sounding but wrong.

Fix: Explicitly write in your topic instructions: "If you cannot find the answer in the available data, respond with: 'I don't have that information right now. Let me connect you with a specialist.' Then invoke the escalation action."

Root Cause 5 Гўв‚¬вЂќ Wrong context passed to LLM

The agent retrieves too much unrelated data (noise) or too little (missing the relevant record), causing the model to either confabulate or miss the answer.

Fix: Tune your Data Cloud vector search query. Use specific filters on data spaces. Limit retrieved chunks to the highest-relevance items. Review the audit trail to see exactly what context was sent to the model on failing turns.

Г°ЕёвЂќВЌ How to Diagnose in Production

Check the audit trail in Data Cloud to see what context was passed to the LLM. Review agent session logs to identify which action returned bad data. Use Agent Builder's conversation preview to replay the exact failing scenario and inspect Atlas's reasoning steps turn by turn.

Module 08

Advanced Design Patterns

Patterns used by senior developers and architects in complex Agentforce deployments.

Judge & Jury

When high accuracy is critical (medical, financial, legal), a second LLM call validates the first agent's response against source data before it's sent. One agent generates, another judges. Prevents hallucinations in high-stakes domains.

RAG Pattern

Retrieval-Augmented Generation. Before generating a response, retrieve relevant documents from Data Cloud vector search and inject them into the prompt. The agent answers from retrieved facts, not general knowledge.

Ensemble Approach

Built into Atlas. Multiple reasoning paths are evaluated in parallel and the most consistent, relevant output is selected. Reduces variance and improves reliability on ambiguous queries.

Headless / Proactive

Agent triggered by a Platform Event or scheduled job Гўв‚¬вЂќ no user conversation. Reads data, executes business logic, updates records, and sends notifications. Best for background enrichment, qualification, and monitoring use cases.

BYO-LLM

Bring Your Own Large Language Model. Connect agents to non-Salesforce models (OpenAI, Azure, Anthropic) via Data 360 infrastructure. Useful when a specialized model outperforms the default for a specific domain.

BYOL

Bring Your Own Lake. Connect agents to data stored in external data lakes (Snowflake, Databricks, AWS S3) via Data 360 federation. Agents can query petabyte-scale data without moving it into Salesforce.
